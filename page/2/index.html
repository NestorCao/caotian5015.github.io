前言

简介

Hadoop Distributed File System (HDFS) is a distributed file system designed to run on commodity hardware. Hardware failure is the norm rather than the exception. detection of faults and quick, automatic recovery from them is a core architectural goal of HDFS.

深入浅出数据库索引原理

发表于 2018-04-09 | 分类于 data

简介

索引是对数据库表中的一列或者多列的值进行排序的一种结构，使用索引可快速访问数据库表中的特定信息。索引的创建大大加快了查询的速度而会因此影响增删改操作的速度。
例如select * from table where id=9999.如果没有索引的话，就会遍历整个表直到id=9999被找到了为止；但是如果建立了索引的话<即对id这一列建立了索引>查询的时候就会现在索引中查找，也就是说索引相当于一个定位系统。打个比方说在我们看小说的时候，往往一本小说要上千页或者更多，如果我们想看到想看的环节，如果一页一页的找那很浪费时间，但是如果有了目录的话，我们直接就可以通过目录来找到，是不是方便了许多。

分布式锁方案

发表于 2018-01-08

简介

我们说单个虚拟机中或者说单个进程中要实现线程安全会采用加锁机制，那分布式的情况下，一定会需要多台虚拟机或者多个进程协调服务，那么多个虚拟机同时操作一块临界资源也会产生线程的安全问题，这就是使用分布式锁的原因。即解决资源在同一时间只能有服务器中的一个线程处理这个资源。

分布式锁的解决方案

1.通过关系型数据库来实现分布式锁，从而保证临界资源的线程安全问题。想象一下，可以在数据库中新增一条记录，谁先占用了这条记录就相当于操作了这块临界资源的锁标记，每个线程在操作时让他先查询数据库的记录，如果有线程存在了就放弃操作或者使用等待轮询的方式继续操作来保证效率，如果发现没有了就直接加锁。但是因为我们操作的是数据库，这相对其他几种方案来说缺点比较大，就比如：使用数据库查询免不了io的一系列操作，所以效率会相对比较低，另一方面一旦数据库当机了，所有的分布式锁就全部失去了效果，也就是说存在了单点故障的问题。
2.通过一些内存型的Nosql实现分布式锁，比如Redis中用setnx这块指令{这个指令的意思就是如果key value存在就会放弃操作，如果key value不存在就会set一个key value来进行保存} 非关系型的数据库相比已第一种方案效率相对较高，采用集群的情况下也不会存在单点故障的问题，并且key value也可以通过设置exprie{时长}从而避免死锁，这种方案相对比较牛逼了，其实相对一第三组方案只是在数据改变是没有通知回馈效果。基本第一种方案凭空想象的应该不会有太大的使用空间
3.通过ZooKeeper临时顺序节点来实现这种分布式锁；原理使用：注册临时节点就加锁 临时节点没了就解锁。

Netty网络通信框架

发表于 2017-09-30 | 分类于 web

简介

Netty是一种异步通讯的网络框架，是用来快速开发可维护行的高性能.搞扩展的协议服务器和客户端，他是封装NIO模型最热门也是最搞笑的网络框架；在Netty里所有的io处理都是异步操作完成的，NIO呢，是JDK1.2版本开始推出的一款同步非阻塞IO模型，而Netty是基于JDK1.7以上开源的封装nio的框架，所以使用Netty的时候JDK版本一定要高于1.6版。

Netty思想

个人感觉Netty这中网络框架相比入Spring Servlet Mybatis等这些在思想上相对复杂抽象些，但是从开发角度讲，基础步骤虽然很墨迹但是完全不需要怎样更改，作为一个开发来说，只需要根据业务有一个完整的业务框架内的一些相关逻辑然后在代码实现上只需对初始化客户端和初始化服务端这两种步骤进行业务更改就可以了{记得导入netty包}；下面从客户端开始说下锁理解的编程步骤。

简单的客户端

在了解这门框架的时候，参照着官网{netty.io}进行开发开源，你会发现基础的步骤不会想差太多，只需要更改里面一个业务逻辑就可以了；说下我理解的客户端netty编程步骤：
1.初始化客户端引导程序，创建客户端的引导对象
2.创建IO处理的线程池，主要进行客户端数据的读写
3.绑定线程池
4.初始化服务通道
5.初始化客户端通道{开发主要在这个位置编写编程}
6.连接服务器，以同步方式等待连接成功
7.同步等待客户端关闭连接
8.释放资源
