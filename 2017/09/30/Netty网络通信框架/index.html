Netty网络通信框架

发表于 2017-09-30

简介

Netty是一种异步通讯的网络框架，是用来快速开发可维护行的高性能.搞扩展的协议服务器和客户端，他是封装NIO模型最热门也是最搞笑的网络框架；在Netty里所有的io处理都是异步操作完成的，NIO呢，是JDK1.2版本开始推出的一款同步非阻塞IO模型，而Netty是基于JDK1.7以上开源的封装nio的框架，所以使用Netty的时候JDK版本一定要高于1.6版。

Netty思想

个人感觉Netty这中网络框架相比入Spring Servlet Mybatis等这些在思想上相对复杂抽象些，但是从开发角度讲，基础步骤虽然很墨迹但是完全不需要怎样更改，作为一个开发来说，只需要根据业务有一个完整的业务框架内的一些相关逻辑然后在代码实现上只需对初始化客户端和初始化服务端这两种步骤进行业务更改就可以了{记得导入netty包}；下面从客户端开始说下锁理解的编程步骤。

简单的客户端

在了解这门框架的时候，参照着官网{netty.io}进行开发开源，你会发现基础的步骤不会想差太多，只需要更改里面一个业务逻辑就可以了；说下我理解的客户端netty编程步骤：
1.初始化客户端引导程序，创建客户端的引导对象
2.创建IO处理的线程池，主要进行客户端数据的读写
3.绑定线程池
4.初始化服务通道
5.初始化客户端通道{开发主要在这个位置编写编程}
6.连接服务器，以同步方式等待连接成功
7.同步等待客户端关闭连接
8.释放资源

object —- buteBuf 编码器 只负责处理出站 —-
—–对象传输协议{处理对象类型}
byteBuf— object 解码器 只负责处理入站 —-
所以在客户端或者服务端单层并不会出现同时编码解码情况{因为编码只负责出站…..，各自分工}
只要处理对象类型的传输协议就要用到编解码器同步使用
编解码之前需要考虑到序列化方案，因为对象的传输需要支持序列化啊！可以用fastjson也可以用jdk原生的序列方式{记得导包}

粘包和拆包

我们常说tcpip协议在传输数据时会发生粘包拆包问题，即会导致数据的丢失和不一致。其实原因很简单，在我们使用协议传输数据的时候，都是以自己数组的形式进行传输，字节数组在我们宏观的思想应该是最小的单元了吧，也就是所谓的‘流’，就是没有分界线的一串数据，但是TCP底层并不知道我们业务数据的具体含义，他会根据缓冲区的实际情况进行包的划分，也就是存包，所以一个完整的数据包可能会被tcp拆分成多个包放在缓冲区进行传输也有可能把多个小的包装到一个缓冲区进行传输，因为数据包的大小随业务而定，但是缓冲区大小是固定的啊，tcp所知道的就是合理的将缓冲区装满，并不会在意你数据包的完整性，所以就发生了所谓的粘包和拆包。<打个比方吧，因为在tcp传输是流的形式，我们已液体为例，当你需要将一些水和一些可口可乐送到外婆家，然后因为条件的限制负责传输的瓶子就一个可以装好这些水和可口可乐，你就会把这两件东西放在一个瓶子里，等你在到外婆家打算去单独和可口可乐的时候，你已经拆不开了，因为可乐里包含了水，水里也有了可乐，这就是粘包了；假如这些水和可乐还是需要送到外婆家，而你手里有两个小瓶子，两个小瓶子可以装下水和可乐，可是水一个瓶子装不下，可乐一个瓶子装完还有余，你会吧可乐装完之后再把一些水放在可乐瓶子里，然后剩下的水就可以装进第二个瓶子，这就是拆包了，不管怎样，因为协议的传输是最小单位的传输，如果将两个或者多个数据同时发送又不指定界限的情况下，就狠容易将数据打乱，就收到的数据就会丢失或者错误>
至于解决策略，有如下几种：
1.消息订场，例如每个报文的大小固定长度200字节，如果不够的话，空位补空格
2.在包尾增加回车换行符{/t /n}进行分割，例如FTP协议{数据传输协议}
3.将消息分为消息头和消息体，消息中记录了消息体的长度
4.采用更复杂的应用层协议
